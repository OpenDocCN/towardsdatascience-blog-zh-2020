答:几乎我们所有人都将开源软件作为生活的一部分。合作创建公共软件已经改变了软件开发的世界和整个世界。但是我们在数据科学领域没有看到同样的行为。**开源数据科学(OSDS)在哪里？**

在宣称开源**数据科学**有意义之前，我们先来解释一下为什么开源**软件**有意义。

# 开源软件的案例

许多人认为 OSS 的好处是显而易见的，大多数开发者和软件公司都与开源社区有某种关系。但并不总是这样。

## 开源软件的(真正的)简史

![](img/da7ff56c3f06f5c5b818ec39c39fffff.png)

很久以前，在一个遥远的星系里。[张家瑜](https://unsplash.com/@danielkcheung?utm_source=unsplash&utm_medium=referral&utm_content=creditCopyText)在 [Unsplash](https://unsplash.com/s/photos/lego?utm_source=unsplash&utm_medium=referral&utm_content=creditCopyText) 上的照片

在一开始(20 世纪 50-60 年代)，几乎所有的软件都是开源的。它是由研究人员和学者创建的，他们本着科学的精神与社区分享它。

但是甚至在更晚的时候，当像 IBM 这样的公司卖出他们的第一台电脑时，[软件与源代码](https://en.wikipedia.org/wiki/History_of_free_and_open-source_software)一起被分发，以使“黑客”能够修复错误并创造改进。

到 20 世纪 60 年代末，封闭软件逐渐成为一种标准，与硬件相比，软件的价格大幅上涨。像许多其他的技术转变一样，[始于对 IBM](https://www.justice.gov/atr/case-document/united-states-memorandum-1969-case) 的反垄断诉讼，其结论是捆绑软件是反竞争的。几年后，在 1974 年的[，美国认为软件是可以被版权保护的](https://digitalcommons.law.ggu.edu/cgi/viewcontent.cgi?referer=https://en.wikipedia.org/&httpsredir=1&article=1344&context=ggulrev)，由此开始了封闭源代码软件的时代(和商业模式)。

尽管开源从来没有消失过(例如像 TeX 这样的项目)，但它经历了一次复兴，始于 1983 年的 GNU 项目，但只是在 20 世纪 90 年代后期才显著显现，出现了像 Linux 和 Git 这样的项目(由同一个人——[Linus Torvalds](https://en.wikipedia.org/wiki/Linus_Torvalds))以及最终的 GitHub。

## 为什么不总是显而易见

有一句关于维基百科的名言同样适用于开源:

> “维基百科的问题在于它只在实践中有效。理论上，这是一场彻底的灾难”——加雷斯·欧文

在开源软件(OSS)的早期，有一种强烈的说法认为软件公司永远不会共享他们的源代码，因为这是他们的知识产权——他们最有价值的财富，把它交给竞争对手没有任何意义。

闭源软件有它的优势——它往往有更好的支持，更丰富的特性集，并且有时更加用户友好。事实上，公司有他们想要保护的知识产权，而且开源软件如何适应开发者或公司的商业模式并不总是清楚的。换句话说，它有时会增加赚钱的难度。

## 开放源码软件的好处

![](img/87688976e4baed5f214275b2a126490d.png)

新的希望。照片由 [Xenomurphy](https://visualhunt.co/a4/48a772c9) 在[visual hunt](https://visualhunt.com/re6/41ae90ab)/[CC BY-NC-ND](http://creativecommons.org/licenses/by-nc-nd/2.0/)

今天，OSS 的许多好处是显而易见的:

*   透明和自由——使用 OSS 意味着你不会被一个供应商所束缚，并且可以很容易地理解你所使用的工具的内部工作原理
*   可靠性和安全性——当有成千上万只眼睛检查您的代码时，识别 bug 和安全风险要容易得多
*   更快的上市时间——有数百名贡献者帮助进行功能开发会快得多。对于那些没有大量员工的创业公司来说尤其如此

然而，有些好处不太直观。例如，在技术行业最困难的事情之一是找到伟大的开发者，他们中的许多人都非常关心 OSS。将 OSS 整合到一个公司的产品组合中可以让开发者展示他们的能力并回馈社区。这对人才有巨大的吸引力。

当然，我们不要拐弯抹角。将一个内部项目作为开源项目发布是很好的 PR(公共关系，而不是拉式请求)，特别是如果你有一个基于它的非开源产品，和/或如果你的用户是开发者。

## 这只是有意义的(在某些情况下)

什么时候创建开源项目最有意义？我认为我们可以将组织的“开源标准”定义如下:

> 当一个项目解决了**公共问题**并且**不是你的核心产品**的一部分时，开源这个项目是一个好主意(尽管这是一个经验法则，也有例外)。

作为例子，我们来看看[的反应](https://github.com/facebook/react)。

![](img/fd8ff0d9f2ba4cf22c1e1d70a3714a5b.png)

帝国在 React 之前设计了他们的第一个用户界面。照片由[张家瑜](https://unsplash.com/@danielkcheung?utm_source=unsplash&utm_medium=referral&utm_content=creditCopyText)在 [Unsplash](https://unsplash.com/s/photos/lego?utm_source=unsplash&utm_medium=referral&utm_content=creditCopyText) 上拍摄

React 是一个用于构建用户界面的 JavaScript 库。它是由脸书在 2013 年发布的。它有麻省理工学院的许可证，这意味着你可以用它做任何你想做的事情，包括商业项目。

现在，从产品角度分析 React，它解决了 web 开发中的一个严重问题——很难从头开始构建用户界面。很明显，这是一个非常普遍的问题，而且解决这个问题也不是 Facebook 作为一家公司的目标——他们 s̶e̶l̶l̶ ̶a̶d̶s̶建立社交网络 [](#3c1c) 。

这意味着 React 是上述标准的一个例子——一个公共问题，而不是公司的核心知识产权。因此，将其作为 OSS 发布保证了脸书的所有好处，而没有太多的负面影响。

# OSDS 的案例

我想现在你应该相信 OSS 是有价值的。现在，让我们来谈谈这些概念是如何转移到数据科学领域的。为了做到这一点，让我们通过给出一个遵守“开源标准”的例子——人脸识别——来描绘 OSDS 如何工作的更详细的愿景。

您可能很容易想到一些可以使用人脸识别的应用程序。对于这个例子，让我们看看在*疾病诊断*和*时尚推荐*的背景下的人脸识别。在谷歌上快速搜索一下，就会发现确实有公司在做这些事情。

在第一个用例中，患者的照片将作为医疗检查的一部分(或在家)被拍摄，然后被分析以识别可能的疾病，帮助医生优先处理紧急情况并治疗更多的人。

在第二个用例中，一个应用程序会根据你的购买历史，以及顾客的面部特征或身体结构来推荐最好的衣服。

这些公司绝对不是*而是*在相互竞争。它们的 IP 在于分别提供用于疾病诊断或时尚推荐的系统。然而，今天，两者很可能正在并行开发他们系统的一个关键部分——人脸识别。这意味着重复工作，浪费大量数据科学家的时间和金钱。

在 OSDS 的世界里，两家公司都将致力于一个开放的人脸检测项目，贡献代码和数据来帮助应对边缘案件，并创建一个更强大的解决方案。与当前状态相比，他们可以让更少的数据科学家来完成这项任务，让他们专注于更棘手、更关键的问题。

想要学习和展示其技能以改进其产品组合的新数据科学家将会投身于识别错误、低效并创建替代模型，从而为不同的用例区分各种指标的优先级。他们可以看到一个严肃的数据科学项目是什么样的。

最重要的是，寻找未来数据科学家的公司可以找到并联系那些已经为他们关心的项目做出贡献的人，从而缩短新团队成员的招聘和入职流程。

最近，人工智能的透明度、多样性和包容性成为科技领域的焦点。OSDS 可以在这些领域产生重大影响。举个例子，假设其中一个产品的用户，碰巧也是一名数据科学家，发现面部检测在她的图像上表现很差。她查看了数据集，发现没有她所属的某个少数民族的图像。她添加了另一个数据集中的图像来填补空白，并提交了一个拉取请求——瞧，新模型对她来说表现得更好了，公司也享受到了一个更加多样化和改进的模型。

真正的双赢局面。

当我开始写这篇文章时，这是一个理论上的例子。现在，情况已经不同了。

为了结束这篇文章，我想写两个例子来说明开源数据科学在实践中应该和不应该是什么样子。

## 伪装成开源数据科学的开源软件

现在，我已经能听到你的反对:“但是我从 GitHub 克隆了 [tensor2tensor](https://github.com/tensorflow/tensor2tensor) 和 [BERT](https://github.com/google-research/bert) ，这已经是开源数据科学了！”或者“我在 [arXiv](https://arxiv.org/list/cs.LG/recent) 上读到的很多论文都在网上发布了代码”。

![](img/afd68bd46b5ed4d08ca7136078e518e7.png)

三位一体——代码、预训练模型和 arXiv 论文。 [GitHub 上的照片](https://GitHub.com)

这不是对 BERT 作者的讽刺，这一部分也不是对这些项目的不尊重。这些项目非常有用，要让公众能够接触到它们需要付出很多努力。他们的作者尽他们所能给出他们所拥有的工具和行业标准。

我想说的是，他们是开放源码软件，但不是 OSDS。

如果一个这样的例子中的代码有一个 bug，这使得发布模型的 API 由于某种原因不能工作，独立的程序员通常可以以 pull 请求的形式贡献一个 bug 修复。另一方面，如果独立的数据科学家意识到数据中存在问题，比如模型对某些种族群体有偏见，在大多数情况下，他们无法修改数据集并创建改进的模型。

这导致对代码的贡献，而不是对项目的数据科学部分的贡献。换句话说，这个项目实际上变成了一个软件项目。

OSDS 还会对数据科学研究产生重要的副作用。

## 提高数据科学研究的质量

提高研究质量是一个值得追求的重要目标。实现这一目标的一些值得注意的努力是:

*   [distilt . pub](http://distill.pub)—一份旨在使机器学习研究更加[可访问、互动和透明的出版物](https://distill.pub/about/)
*   [开放评审](https://openreview.net/) —一个开放的科学交流平台，致力于使同行评审过程透明化
*   [arXiv](https://arxiv.org/) —在完全同行评审之前[被批准发布的论文的电子印刷库](https://en.wikipedia.org/wiki/ArXiv)
*   [代码为](https://paperswithcode.com/)的论文——机器学习研究论文的在线集合，结合了它们的代码，简化了实现。

关于数据科学研究中一些最新(SOTA)成果的问题本质，已经说了很多。通常，当一篇论文发表后，你只能看到最终结果，即与之前的 SOTA 相比提高的百分比。很多时候，小的改进可能归因于随机种子的成功选择或执行许多实验，直到一个人成功地改进了一个指标，而没有充分考虑结果的[统计显著性](https://en.wikipedia.org/wiki/Statistical_significance)。

更简单地说，由于研究过程不透明，我们不知道一个结果实际上代表了研究的进步还是运气的侥幸。

在 OSDS 的世界里，研究团队可以发布他们的整个研究历史，同时提交论文以供审查。审稿人将能够提供高质量的反馈，并确保严谨的研究方法。每个人都将享受更高质量的数据科学研究。
